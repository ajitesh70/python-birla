# SonarQube SAST — What & Why

## What is SonarQube

SonarQube is a self-hosted static code analysis platform. It reads source code
(no build/deploy required for Python) and reports on code quality and
security issues, then enforces a **Quality Gate** — a pass/fail check the CI
pipeline honors before allowing a build to proceed.

Our instance: `https://sonar.tyagi.fun`. This project's key: `bajaj-python`.

## Why we need it

- Catches real bugs, security vulnerabilities, and maintainability problems
  automatically, on every push/PR — before a human reviewer has to find them.
- Enforces a hard gate: if the code doesn't meet the bar, the pipeline stops
  before the Docker image is ever built or pushed (`build-and-push` in
  `ci.yml` only runs if the `quality` job, including the Sonar gate, passes).
- Turns "code review opinions" into consistent, automated, repeatable checks.

## What it scans for

| Category | What it means | Blocks the gate? |
|---|---|---|
| **Bugs** | Code that's likely to behave incorrectly (logic errors) | Yes |
| **Vulnerabilities** | Confirmed security flaws (e.g. injection) | Yes |
| **Security Hotspots** | Security-sensitive code that needs human review to confirm safe/unsafe (e.g. hardcoded secrets, weak crypto, debug flags) | Yes, until reviewed |
| **Code Smells** | Maintainability issues (dead code, duplication of literals, overly complex functions, etc.) | Yes |
| **Duplications** | Copy-pasted code instead of reused | Yes |
| **Coverage** | How much of the *new/changed* code is exercised by tests (from `coverage.xml`, generated by pytest-cov) | Yes |

Bugs/Vulnerabilities/Code Smells all fall under **Issues**; Security Hotspots
are tracked separately and require manual triage (mark Safe/Fixed) in the
SonarQube UI rather than being auto-resolved by code changes alone.

## Metrics we've implemented (Quality Gate — "Sonar way" default)

The gate evaluates only **new code** (changes since the previous analysis),
not the whole legacy codebase:

| Condition | Threshold |
|---|---|
| New code coverage | ≥ 80% |
| New duplicated lines | ≤ 3% |
| New Security Hotspots reviewed | 100% |
| New Bugs/Vulnerabilities/Code Smells (violations) | 0 |

Config lives in [`sonar-project.properties`](sonar-project.properties) and
the `SonarQube Scan` step in
[`.github/workflows/ci.yml`](.github/workflows/ci.yml): sources (`.`),
exclusions (`venv/**`, `__pycache__/**`, `tests/**`, `.github/**`), and the
coverage/test-result report paths (`coverage.xml`, `test-results.xml`).

## Negative testing we've done

To confirm the pipeline actually catches problems (not just a green rubber
stamp), we deliberately committed known-bad code, clearly marked
`# SAST TEST` in `app.py`, and watched the gate react:

**Round 1 — security issues** (SQL injection, OS command injection, MD5
password hashing, insecure `random` token generation, a hardcoded DB
password, Flask debug mode enabled): all landed as **Security Hotspots**, not
Issues. Gate failed on `new_security_hotspots_reviewed` (0% reviewed).

**Round 2 — bugs, code smells, duplication** (mutable default argument,
`if x == x` identical-expression bug, bare `except:`, unused variable, a
leftover TODO comment, and a byte-for-byte duplicate of an existing
function): SonarQube reported **1 Bug**, **6 Code Smells**, and **15.7%**
duplication on new code (threshold 3%). It also caught one *unplanted* smell
— a string literal (`"item not found"`) that became a 3rd duplicate once we
added a new use of it — confirming it's genuinely analyzing the diff, not
just matching what we labeled.

In both rounds: `quality` job failed as expected, and `build-and-push`
never ran — proof the gate actually blocks non-compliant code from shipping,
not just reports on it after the fact.
